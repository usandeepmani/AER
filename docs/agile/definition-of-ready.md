# ✅ Definition of Ready (DoR)

A task is considered **Ready** when it is sufficiently defined and can be started without additional planning.

---

# General Criteria

Before moving a task to **🚧 In Progress**, ensure that:

- [ ] The objective is clear.
- [ ] The expected outcome is defined.
- [ ] The required learning resource(s) are identified.
- [ ] The task type is assigned.
- [ ] The Epic is assigned.
- [ ] The Sprint (Iteration) is assigned.
- [ ] The estimated effort is added.
- [ ] Any prerequisites are completed.

---

# Task Type Requirements

## 📚 Learning

Ready when:

- [ ] Resource is selected.
- [ ] Learning objective is defined.
- [ ] Expected artifact is identified.

Expected artifacts:

- Notes
- Code example (optional)

---

## 💻 Coding

Ready when:

- [ ] Problem statement is understood.
- [ ] Development environment is ready.
- [ ] Acceptance criteria are defined.

Expected artifacts:

- Source code
- Git commit

---

## 🚀 Project

Ready when:

- [ ] Project scope is defined.
- [ ] Success criteria are agreed.
- [ ] Required technologies are known.

Expected artifacts:

- Project folder
- README
- Documentation

---

## 🔍 Review

Ready when:

- [ ] Previous work has been completed.
- [ ] Notes and code are available for review.

Expected artifacts:

- Retrospective
- Improvement actions

---

# AER Principle

> A task should never enter **In Progress** if its objective or expected artifact is unclear.
